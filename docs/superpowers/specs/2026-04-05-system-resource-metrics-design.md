# 系统资源占用接口设计

## 背景

当前仓库 `gmcc` 已提供账号、实例、集群状态和认证相关的 HTTP API，前端已经可以基于 `GET /api/status`、`GET /api/accounts`、`GET /api/instances` 做第一版管理面板。

但现有 API 还没有宿主机系统资源占用信息，前端如果要做首页总览或运行状态卡片，缺少 CPU 和内存这类基础指标来源。

本次设计目标是在不引入复杂监控体系的前提下，新增一个简单、稳定、跨平台的系统资源接口，供前端面板直接使用。

## 目标

- 新增一个系统资源占用接口，返回宿主机 CPU 和内存快照。
- 支持 Windows、Linux、macOS。
- 采用成熟第三方跨平台库，避免手写平台分支采集逻辑。
- 保持现有 Web 层风格，错误处理与当前 API 语义一致。
- 为前端首页提供可直接消费的数据结构。

## 非目标

- 本次不采集磁盘、网络、温度、进程列表等更多系统指标。
- 本次不采集 gmcc 进程级 CPU / 内存占用。
- 本次不采集每个实例的资源占用。
- 本次不引入长期后台采样、缓存、时序数据库或监控子系统。
- 本次不扩展为一整套 `/api/system/*` 接口族。

## 方案对比

### 方案 A：新增独立接口 `GET /api/resources`（推荐）

- 单独暴露一个资源快照接口。
- 返回宿主机 CPU 和内存当前值。

优点：

- 不污染现有 `GET /api/status` 的集群语义。
- 路由短且直观，不为了未来扩展过早增加层级。
- 前端首页可以单独轮询，不影响其他状态接口。

缺点：

- 前端首页需要额外请求一个接口。

### 方案 B：把系统资源并入 `GET /api/status`

- 在现有状态接口中追加 CPU / 内存字段。

优点：

- 前端少调一个接口。

缺点：

- 混合“集群运行状态”和“宿主机资源占用”两个维度，响应结构会越来越杂。
- 对已经依赖 `GET /api/status` 的前端更有侵入性。

### 方案 C：设计为 `GET /api/system/resources`

- 为系统类接口预留一个命名空间。

优点：

- 命名上更规整，便于未来扩展 `system` 下的更多接口。

缺点：

- 当前只有一个资源接口，这一层级是空的，属于过早设计。

### 结论

采用方案 A，即新增 `GET /api/resources`。

这样最符合当前需求，也最符合仓库现有“够用即可、边界清晰”的风格。

## 接口设计

### 路由

- `GET /api/resources`

### 成功响应

```json
{
  "cpu_percent": 12.5,
  "memory": {
    "total_bytes": 17179869184,
    "used_bytes": 8589934592,
    "available_bytes": 7516192768,
    "used_percent": 50.0
  },
  "collected_at": "2026-04-05T12:00:00Z"
}
```

### 字段说明

- `cpu_percent`
  - 宿主机整体 CPU 使用率，范围预期为 `0-100`
- `memory.total_bytes`
  - 总内存字节数
- `memory.used_bytes`
  - 已使用内存字节数
- `memory.available_bytes`
  - 当前可用内存字节数
- `memory.used_percent`
  - 内存使用率，范围预期为 `0-100`
- `collected_at`
  - 采样时间，固定使用 UTC `RFC3339` 格式输出

## 错误语义

- `503`
  - 资源采集器未初始化
  - 响应：`{"error":"resource metrics collector not initialized"}`
- `500`
  - 资源采集失败
  - 响应：`{"error":"failed to collect system resources"}`
- `200`
  - 成功返回资源快照

设计原则：

- 不把底层平台错误细节直接暴露给前端
- 详细失败原因保留在服务端日志中
- 对前端只返回稳定、可消费的错误语义

## 架构与组件边界

### 1. 系统资源采集层：`internal/systemmetrics`

职责：

- 采集宿主机 CPU 和内存指标
- 屏蔽第三方库与平台差异
- 向上层返回统一快照结构

建议对外暴露一个小接口，例如：

- `Collect() (Snapshot, error)`

其中：

- `Snapshot` 由 `internal/systemmetrics` 定义并导出
- Web 层依赖采集器接口和 `systemmetrics.Snapshot`，不依赖第三方库类型
- 默认实现可通过构造函数注入到 Web 层，测试中可替换为 stub / fake 实现

边界：

- 不依赖 Gin 或 Web 层
- 不负责 HTTP 状态码和 JSON 响应
- 不关心前端刷新策略

### 2. Web 层：`internal/web`

职责：

- 注册 `GET /api/resources`
- 检查采集器是否存在
- 调用采集器获取快照并返回 JSON
- 将采集失败转换为稳定的 HTTP 错误响应

边界：

- 不直接调用平台 API
- 不自己计算 CPU / 内存
- 只依赖采集器接口

### 3. 响应模型层：`internal/webtypes`

职责：

- 定义该接口返回给前端的 JSON 结构
- 保持字段命名和现有 API 风格一致

### 4. 启动组装层：`cmd/gmcc/main.go`

职责：

- 初始化默认系统资源采集器
- 将采集器注入 `web.NewServer(...)`

边界：

- 不承担采集逻辑本身
- 不在 `main` 内直接进行系统指标读取

## 第三方库策略

本次明确允许使用成熟第三方跨平台库实现系统指标采集。

推荐方向：使用 `gopsutil` 这一类跨平台库。

原因：

- Windows、Linux、macOS 的系统指标来源差异较大
- 手写平台分支会显著增加复杂度和维护成本
- 当前需求只是返回 CPU + 内存快照，不需要自己实现底层平台适配

约束：

- 只引入本次需求真正需要的库能力
- 不围绕该库建立额外复杂封装层
- 上层代码仍只依赖本仓库内部接口，而不是把第三方库类型直接暴露到 Web 层

## 采样策略

第一版采用“请求时即时采样”模式：

- 每次调用 `GET /api/resources` 时，现场采集一份 CPU / 内存快照
- 不提前做后台周期采样
- 不新增采样缓存、锁或额外 goroutine

原因：

- 实现最简单，易于验证
- 对管理面板类前端已足够实用
- 更符合当前仓库偏好“显式、稳定、可预测”的实现风格

如果后续发现前端轮询频率高，或者 CPU 百分比采样策略需要更平滑，再单独设计缓存采样方案。

CPU 百分比在本次设计中的约束如下：

- 必须表示“整机归一化后的总体 CPU 使用率”
- 返回范围固定为 `0-100`
- 不允许返回多核累加值
- 允许为获得稳定值而在单次请求内使用一个很短的采样窗口
- 允许接口因此产生轻微额外延迟，但不引入后台缓存采样机制

## CPU 指标定义

本次只返回“宿主机整体 CPU 使用率”：

- 不返回每核心使用率
- 不返回 load average
- 不返回进程级 CPU 占用

这一定义是强约束，不允许实现返回“每核心求和后的百分比”或其他前端无法直接展示的语义。

目标是为前端提供一个简单总览值，而不是做系统监控平台。

## 内存指标定义

本次只返回：

- 总内存
- 已用内存
- 可用内存
- 内存使用率

不返回：

- swap
- cache / buffers 细分
- NUMA 或其他平台专有信息

## 跨平台行为

目标平台：

- Windows
- Linux
- macOS

设计要求：

- 通过同一个内部采集接口对外提供统一结果
- 平台差异只存在于第三方库内部或采集层内部
- Web 层与前端都不感知平台差异

如果采集层在当前平台整体无法提供 CPU / 内存快照，应返回错误，由 Web 层转为 `500`。

失败策略固定为：

- CPU 和内存任一关键指标采集失败，接口整体返回 `500`
- 不做“部分字段成功、部分字段缺失”的降级响应
- 这样可以保证前端拿到的响应要么是完整快照，要么是明确失败

## Web 集成约束

- 该接口与现有其他 `/api` 读接口一样，挂在同一组 API 路由下
- 沿用当前已有中间件行为，例如现有 CORS 处理
- 当前仓库默认不启用 Web 密码认证，因此该接口也保持无额外鉴权
- 该接口是读操作，不纳入 `logOperation` 审计日志记录范围

## 测试策略

至少补充以下测试：

- `internal/systemmetrics` 包测试：
  - 采集结果结构完整
  - 采集失败路径可被测试
- `internal/web` 包测试：
  - `GET /api/resources` 成功返回 200
  - 采集器未初始化时返回 503
  - 采集失败时返回 500
- 断言口径：
  - `cpu_percent` 只断言可解析且范围在 `0-100`
  - 内存字段只断言存在、类型正确、并满足基本大小关系
  - `collected_at` 只断言可解析为 UTC `RFC3339` 时间，不断言具体值
- 文档更新：
  - `docs/api.md` 包含该接口的请求路径、响应示例和错误语义

## 成功标准

- 新增 `GET /api/resources`
- 前端可通过该接口直接获取 CPU 和内存快照
- Windows、Linux、macOS 均走统一接口设计
- 使用第三方跨平台库而不是手写平台分支
- Web 层与采集层边界清晰
- 有对应测试覆盖成功与失败路径
- 文档已更新，前端可直接对接
