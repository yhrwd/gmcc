# gmcc 冒险基地前端设计

## 背景

`gmcc` 当前已经具备完整的后端 API，覆盖集群状态、宿主机资源、账号管理、Microsoft 设备码登录、实例管理与操作日志。仓库中也已经为前端嵌入流程预留了 `frontend/` 与 `internal/webui/dist/` 的交付路径，因此现在的工作重点不再是讨论“是否支持前端”，而是为这个 API 体系设计一套真正可用、且有鲜明个性的管理界面。

这个前端不是传统严肃的企业后台。用户希望它“好看优雅，有创意”，同时更偏“可爱 + 高级混合”，并在最后明确希望它“再好玩一点，不需要太专业，功能在就行”。因此本次设计的核心，不是最大化运维感，而是在保证主要操作路径清晰的前提下，把它做成一个带世界观气质的 Minecraft 集群账号管理控制台。

## 目标

- 设计一套完整的前端后台骨架，至少覆盖总览、账号管理、实例管理、资源监控、操作日志五个工作区。
- 视觉上实现“可爱冒险基地风 + 高级材质 + 清晰管理流”的统一体验。
- 让首页同时承担监控和运营入口，采用平衡型信息布局，而不是只偏资源监控或只偏账号运营。
- 让 Microsoft 设备码登录、账号状态判断、实例启停等核心操作更轻松、更有趣，但仍保持可理解与可执行。
- 前端与现有 `/api` 路径、嵌入式交付方式、Vite 构建产物约束兼容，不改变后端既有接口语义。

## 非目标

- 本次不追求传统企业级控制台的严肃视觉语气。
- 本次不要求一次性做完整复杂的数据分析、权限系统或高级筛选体系。
- 本次不改变现有 Go 服务 API 语义，不增加新的后端接口前提作为设计成立条件。
- 本次不把界面做成纯游戏启动器；它仍然是管理面板，只是表达方式更有趣。

## 已确认的用户偏好

- 风格：`可爱 + 高级混合`
- 范围：`整套后台骨架`
- 首页信息重心：`平衡型`
- 最终调性：`再好玩一点，不需要太专业，功能在就行`

这些偏好意味着设计不应走纯奶油萌系，也不应走冷硬赛博运维风，而应在高级材质和轻松叙事之间取平衡。

## 方案对比

### 方案 A：云巢控制台

特点：奶油杏、暖白、柔和玻璃感、大圆角、治愈型视觉氛围。

优点：

- 亲和力最强，容易做出“可爱”体验。
- 对账号卡片、空状态、轻提醒类模块非常友好。

缺点：

- 如果大量用于实例控制和资源监控，容易显得偏甜。
- 不利于承接“集群控制台”这一产品身份。

### 方案 B：棱镜栖居舱

特点：雾蓝、薄荷、月白、轻玻璃与透光层次，局部加入暖色和像素细节。

优点：

- 最适合账号管理、实例状态、资源监控共存的产品场景。
- 容易在“高级”和“可爱”之间建立稳定平衡。
- 可以通过组件级彩蛋提升趣味，而不破坏整体结构。

缺点：

- 如果处理过于克制，会偏回普通现代控制台。

### 方案 C：公会观星台

特点：叙事感更强，更像世界观后台、冒险者基地或公会指挥面板。

优点：

- 记忆点很强，视觉表达最有故事性。
- 很适合日志、账号角色化、事件流式呈现。

缺点：

- 如果比重过高，会削弱管理面板的效率感。
- 更容易让界面向“游戏 UI”倾斜，而不是管理台。

### 结论

采用 `方案 B 为主 + 方案 A 的暖感 + 方案 C 的少量叙事点缀`。

最终方向定义为：`可爱冒险基地风 + 高级材质 + 清晰管理流`。

这意味着底层界面结构与信息组织仍然稳定、现代，但在标题、空状态、图标、动效、状态徽章和局部版式上，让产品看起来更像一个“冒险基地控制台”，而不是传统企业后台。

## 产品定位与体验原则

### 1. 产品定位

前端定位为“MC 集群账号与实例管理面板”，不是数据分析平台，也不是纯启动器。它需要同时承担三类职责：

- 看状态：集群状态、资源占用、实例存活情况
- 管账号：创建账号、查看账号状态、发起 Microsoft 设备码登录
- 管实例：创建实例、启动、停止、重启、查看实例摘要

### 2. 体验原则

- 优先让用户感到“有趣但不迷路”。
- 关键操作必须一眼可见，不藏在多层交互里。
- 情绪表达通过文案、插画、徽章、材质和动效完成，而不是通过复杂游戏化机制完成。
- 即使界面更俏皮，数据和状态仍然要可扫读。

### 3. 文案原则

- 模块标题与导航可偏稳重，例如“总览”“账号管理”“实例管理”。
- 空状态、次级提示和状态反馈可以更有趣，例如：
  - “还没有冒险者入驻”
  - “这只账号还没起床”
  - “基地里还没有出勤的小家伙”
- 成功、失败和风控类提示仍应直白，不牺牲可理解性。

## 信息架构

第一版前端采用完整后台骨架，包含以下一级区域：

- `总览`
- `账号管理`
- `实例管理`
- `资源监控`
- `操作日志`

导航优先考虑左侧纵向主导航，移动端退化为底部标签或抽屉式导航。

### 各区域职责

#### 总览

- 展示集群总体状态
- 展示 CPU / 内存等资源摘要
- 展示运行中实例概况
- 展示待登录账号或异常状态提示
- 展示最近操作动态

#### 账号管理

- 展示账号列表与状态
- 支持创建账号
- 支持发起 Microsoft 设备码登录
- 让用户快速理解哪个账号可用、哪个账号仍待认证

#### 实例管理

- 展示实例列表与运行状态
- 支持创建实例
- 支持启动、停止、重启
- 在卡片中展示 server address、在线时长、基础生存状态等信息

#### 资源监控

- 展示宿主机 CPU 和内存资源
- 强调“现在是否健康”，而不是复杂历史分析
- 允许作为总览页的补充详情区存在

#### 操作日志

- 展示操作审计与最近动态
- 默认使用更轻松的时间线 / 事件流样式
- 如果需要高密度信息，可以保留紧凑视图或列表视图切换

## 首页布局设计

首页采用用户确认的 `平衡型` 结构，同时承载监控与运营入口。

### 第一屏结构

- 顶部：全局状态横幅
- 中部左侧：基地总览主卡
- 中部右侧：账号工作区 / 待办工作区
- 底部左侧：实例运行卡片区
- 底部右侧：基地广播栏（最近日志）

### 顶部全局状态横幅

信息来源主要对应 `GET /api/status` 与 `GET /api/resources`。

展示内容：

- `cluster_status`
- `total_instances`
- `running_instances`
- 当前 CPU 使用率
- 当前内存使用率
- 最近采样时间

表现方式：

- 像“基地今日状态”的头部横幅，而不是硬性告警栏
- 状态标签应有明确色彩区分：运行、部分运行、停止
- 资源压力高时可以增强描边或发光，而不是直接把页面变得很红很硬

### 基地总览主卡

主卡承担页面情绪建立与核心事实归纳：

- 当前有多少实例正在出勤
- 资源是否处于舒适区
- 是否存在待认证账号
- 最近是否有重要状态变化

主卡可以使用更有世界观感的标题，例如“今日基地状态”或“冒险基地正在运作中”，但正文数据必须清楚。

首页摘要推导规则：

- “待认证账号”取自待处理账号集合
- “最近登录成功的账号”最多展示 1 到 3 个已登录账号
- “是否有重要状态变化”第一版以最近日志是否为空为近似信号，不单独构建复杂事件检测器
- “基地是否舒适”仅由当前资源快照与集群状态联合得出
- “实例是否活跃”优先取 `running_instances` 与实例列表中的运行态摘要

### 账号工作区

放置：

- 待登录账号
- 最近登录成功的账号
- 快速发起 Microsoft 设备码登录入口
- 需要处理的账号状态提示

该区域更偏“运营工作台”，帮助用户在进入账号管理页之前就完成关键动作。

### 实例运行区

实例以卡片而不是密集表格为主，重点展示：

- 实例 ID
- 绑定账号 ID
- `running/stopped/unknown` 状态
- `server_address`
- `online_duration`
- `health`、`food`
- 可选位置摘要

卡片需要支持直接触发 `start / stop / restart`。

### 基地广播栏

来源对应 `GET /api/logs/operations`。

采用时间线或事件流式呈现，使“操作日志”更像基地动态，而不是传统后台流水。它可以使用更柔和的叙事文案与图标，但需要保留时间、动作、结果等关键信息。

## 视觉系统

### 配色

主配色建议：

- 雾蓝：作为主要结构与大面积基底色
- 薄荷绿：作为运行态、活跃态与成功态强调色
- 月白：作为高亮表面与半透明层
- 奶油杏：作为温度色，用在提示、徽章或重点交互上
- 星砂金：作为很少量的精致强调色

避免事项：

- 不使用高饱和粉色大面积铺底
- 不使用偏俗套的蓝紫赛博风主配色
- 不使用大面积纯黑高压对比

### 材质

- 页面背景采用浅雾渐变、低对比漫反射形状或轻纹理，而不是纯平背景
- 卡片采用柔玻璃、微描边、软阴影
- 局部加入方块切角、像素角标、徽章、贴纸式标签，形成 MC 气质

### 字体与图形语言

- 正文和数据区保持现代、清晰的无衬线体系
- 标题或局部装饰可带一点像素感，但不能影响信息密度和阅读效率
- 图标建议以圆润几何图标为主，局部融合像素元素

## 交互设计

### 1. 账号创建与登录

账号创建应保持轻量：

- 打开简单创建面板
- 输入账号必要信息
- 创建后立即引导到登录动作

第一版账号创建表单最小字段契约：

- `id`：必填，对应 `POST /api/accounts` 的账号标识；提交前做 trim，不能为空
- `label`：可选；为空时允许提交
- `note`：可选；为空时允许提交

校验与提交规则：

- `id` 为空时前端直接阻止提交，并给出明确提示
- 提交中禁用提交按钮，防止重复创建
- 创建成功后关闭创建面板、刷新账号列表，并将新账号置于显著位置
- 创建成功后应直接提供“继续登录”入口
- 创建失败时保留用户输入，并显示接口返回的错误信息

Microsoft 设备码登录流程应该被做成一个“唤醒账号”的任务面板，而不是生硬的技术弹窗。面板至少展示：

- 账号 ID
- `user_code`
- 复制按钮
- 前往 `verification_uri_complete` 的明确按钮
- 轮询中状态
- 成功 / 失败反馈

由于当前后端采用单活跃设备码会话模型，前端必须让用户清楚地知道“新的发起动作会替换上一轮会话”。

#### 设备码登录状态机

第一版登录面板至少包含以下状态：

- `idle`：尚未发起登录
- `initializing`：正在请求设备码
- `polling`：设备码已生成，正在轮询
- `succeeded`：登录成功
- `failed`：登录失败
- `expired`：设备码过期
- `replaced`：被新的登录会话替换

面板显隐状态独立于任务状态：

- `panelOpen=true`：显示登录面板
- `panelOpen=false`：隐藏登录面板

关闭面板不会把任务状态改写为新的任务枚举值。

状态契约：

- 同一时刻只允许一个活跃轮询任务
- 当用户对另一个账号重新发起登录时，旧任务立刻停止轮询并进入 `replaced`
- 第一版不要求二次确认，可直接替换，但必须给出明显提示
- 被替换的旧面板若仍可见，应显示“当前登录任务已被新的会话替换”
- 用户关闭面板只关闭前端视图，不代表后端会话一定失效，因此再次打开时以前端内存中的当前已知状态恢复显示
- 轮询中若出现短暂网络错误，不立即判定失败，应允许有限次重试或给出继续重试入口

轮询策略：

- 轮询节奏优先使用接口返回的 `interval`
- 遇到瞬时网络失败时，最多连续容忍 3 次失败，期间维持 `polling` 并显示连接波动提示
- 连续超过 3 次网络失败后，进入 `failed`，并提供“继续重试”按钮
- 用户点击“继续重试”时，复用当前面板上下文重新进入轮询，不自动新建第二个会话
- 一旦进入 `replaced`，必须立即终止旧轮询，不再自行恢复

状态来源与恢复规则：

- 第一版只要求在单次前端会话内恢复登录面板状态
- 用户切换页面再返回时，应保留当前登录面板状态
- 用户关闭登录面板后重新打开时，应恢复该前端会话中最后一次已知状态
- 浏览器整页刷新后，不要求恢复旧登录任务；第一版可回到 `idle`
- 第一版不引入额外后端查询接口来重建设备码中间状态
- `succeeded` 由轮询成功响应驱动
- `failed` 由明确失败响应或连续网络失败超限驱动
- `expired` 由轮询结果明确指向过期语义时驱动
- `replaced` 只由前端发起新的登录任务时驱动

状态归属：

- 设备码登录任务属于应用级单例状态
- 首页快捷登录入口与账号页登录入口共享同一任务状态
- 任一入口发起新的登录任务时，都要覆盖旧任务并同步更新所有可见入口

### 2. 账号状态表达

账号至少需要清晰区分：

- 已登录
- 待登录
- 禁用
- 无 token / 不可用

状态标签既要明确，也要有点个性。例如已登录可以是柔亮的“在线冒险者”，未登录可以是“待唤醒”，但底层语义仍需与真实 `auth_status` 对齐。

账号状态映射规则：

- `enabled=false`：优先显示为“禁用”，不再强调其他登录语义
- `auth_status=logged_in` 且 `has_token=true`：显示为“已登录”
- `auth_status` 表示待认证、未认证或 `has_token=false`：显示为“待登录”
- `auth_status` 明确表示认证失效或不可用：显示为“不可用”
- 无法识别的 `auth_status`：显示为“状态未识别”

前端推导规则：

- “待处理账号”指当前未禁用且不能直接用于实例创建的账号
- “最近登录成功的账号”优先从当前列表中选取状态为“已登录”的账号展示，第一版不额外追踪历史成功时间

### 3. 实例控制

实例卡片上直接提供：

- 启动
- 停止
- 重启

按钮组应放在易发现的位置，不要求进入详情页才能操作。

运行中实例卡可以有轻微呼吸感高亮；停止态实例卡则更安静、雾化。动效要小，不要喧宾夺主。

实例动作边界：

- 单张实例卡在写操作进行中时，应锁定当前动作按钮，防止重复提交
- 若操作成功但列表摘要尚未刷新，先展示临时成功反馈，再触发局部刷新
- 若操作失败，应恢复按钮可用状态并显示明确错误原因
- 第一版不要求乐观更新到最终状态，以接口刷新结果为准

第一版实例创建表单最小字段契约：

- `id`：必填，提交前做 trim，不能为空
- `account_id`：必填，必须从当前账号列表中选择
- `server_address`：必填，提交前做 trim，不能为空
- `enabled`：可选，默认 `true`
- `auto_start`：可选，默认 `false`

校验与提交规则：

- 若 `account_id` 未选择，则阻止提交
- 若 `enabled=false` 且 `auto_start=true`，前端直接阻止提交
- 提交成功后关闭创建面板、刷新实例列表，并在实例区高亮新创建项
- 若账号当前不满足创建实例条件，应直接显示失败原因，不做隐式兜底

### 4. 空状态设计

空状态是本设计中重要的个性表达点。页面或模块无数据时，不应只是“暂无数据”，而应该兼顾功能指引与情绪表达。例如：

- 无账号：引导先创建账号
- 无实例：引导先完成账号登录，再创建实例
- 无日志：提示基地还很安静

空状态可以带小插画或像素生物元素，但必须配合一个清晰的下一步按钮。

## 数据映射与页面内容边界

本次设计建立在现有 API 上，不要求新增后端接口才能成立。

### 首页聚合加载策略

首页属于多接口聚合页，默认并行请求以下接口：

- `GET /api/status`
- `GET /api/resources`
- `GET /api/accounts`
- `GET /api/instances`
- `GET /api/logs/operations`

加载与降级规则：

- 首屏允许各模块独立进入 loading，不要求等待所有接口完成后再整页显示
- 任一接口成功后，对应模块即可先显示内容
- 任一接口失败后，只让对应模块显示错误态与重试入口，不阻塞其他模块
- 首页保留一个“全部刷新”入口，同时每个模块允许局部重试
- 自动刷新只覆盖状态型数据：`status`、`resources`、`accounts`、`instances`
- `logs/operations` 第一版默认手动刷新或低频刷新，不要求高频轮询

部分失败时的展示规则：

- `status` 失败：顶部状态横幅改为“状态获取失败”，但其他卡片仍可显示
- `resources` 失败：资源卡展示不可用提示，不影响账号与实例模块
- `accounts` 失败：账号工作区进入错误态，隐藏依赖账号列表的快捷操作
- `instances` 失败：实例区进入错误态，不影响总览主卡和日志区
- `logs/operations` 失败：广播栏显示“暂时收不到基地动态”，并保留重试按钮

刷新策略：

- 页面初始化时执行一次全量加载
- 用户触发关键写操作成功后，刷新受影响模块，而不是机械地全页重载
- 自动刷新频率在实现阶段可保持温和，目标是“状态可感知”而不是实时监控墙

### 首页与总览页主要依赖

- `GET /api/status`
- `GET /api/resources`
- `GET /api/accounts`
- `GET /api/instances`
- `GET /api/logs/operations`

### 账号管理页主要依赖

- `GET /api/accounts`
- `POST /api/accounts`
- `POST /api/auth/microsoft/init`
- `POST /api/auth/microsoft/poll`

### 实例管理页主要依赖

- `GET /api/instances`
- `POST /api/instances`
- `POST /api/instances/:id/start`
- `POST /api/instances/:id/stop`
- `POST /api/instances/:id/restart`

### 设计约束

- 所有前端请求使用同源相对路径 `/api`
- 第一版可优先依赖列表接口填充详情卡片，不强依赖单项详情接口
- 日志默认先服务于“最近事件流”，不强制做复杂筛选器
- 资源监控先看当前快照，不预设完整时序图表系统

### 页面字段映射与缺失值规则

#### 实例状态映射

- `running`：实例正常运行；允许动作为 `stop`、`restart`
- `stopped`：实例当前停止；允许动作为 `start`
- `partial`：集群整体层面的状态词，第一版不作为实例卡片专属状态枚举来源
- `unknown`：实例 `status` 缺失或无法识别时使用；允许显示中性样式，并保留谨慎动作入口

实例卡片中的状态展示规则：

- 若接口返回的是可识别实例状态，则按实例状态渲染
- 若无法识别，则降级为 `unknown`
- 第一版不假设实例接口一定返回 `partial`
- `unknown` 状态下不使用运行态高亮视觉
- `unknown` 状态下默认允许 `restart`，但不突出展示 `start` 或 `stop` 的强状态语义

#### 顶部状态横幅

- `cluster_status`：来自 `GET /api/status`，必须有值；若接口失败则整块进入错误态
- `total_instances`：来自 `GET /api/status`，缺失时显示 `--`
- `running_instances`：来自 `GET /api/status`，缺失时显示 `--`
- CPU / 内存：来自 `GET /api/resources`，接口失败时分别显示“不可用”
- 最近采样时间：来自 `resources.collected_at`，缺失时显示“刚刚无法采样”

#### 账号工作区

- `id`、`label`、`note`、`enabled`、`auth_status`、`has_token`：来自 `GET /api/accounts`
- 若 `label` 为空，则回退显示 `id`
- 若 `note` 为空，则不保留占位段落
- `auth_status` 未知或缺失时，统一归为“状态未识别”，使用中性样式
- 第一版不假设额外账号扩展字段存在

#### 实例运行区

- `id`、`account_id`、`server_address`、`status`、`online_duration`、`health`、`food`、`position`：来自 `GET /api/instances`
- `status` 缺失时，按“未知”处理，不渲染启停态专属视觉
- `server_address` 为空时，显示“未填写地址”
- `online_duration` 缺失时，显示“刚刚开始记录”
- `health`、`food` 缺失时，不显示对应状态条，而不是伪造 0 值
- `position` 缺失时，不显示位置摘要模块

#### 日志广播栏

- 第一版只要求读取最近日志摘要，不要求完整高级筛选体系
- 若日志接口成功但为空，显示“基地还很安静”空状态
- 若日志接口失败，显示错误卡并允许局部重试

#### 前端推导字段

- “待处理账号数”由 `accounts` 列表中不可直接用于实例的账号数量推导
- “运行中实例占比”由 `running_instances / total_instances` 推导
- “基地是否繁忙”由状态与资源联合得出，仅用于视觉描述，不作为业务判断依据

### 第一版必须项 / 可延期项

#### Must

- 完整应用骨架与导航
- 首页平衡型布局
- 账号列表、创建账号、发起 Microsoft 设备码登录、轮询结果反馈
- 实例列表、创建实例、启动、停止、重启
- 宿主机资源卡与集群状态摘要
- 最近日志时间线或事件流
- 桌面与移动端基础可用
- 关键错误态、空状态、成功反馈

第一版不包含：

- 删除账号流程
- 删除实例流程

#### Should

- 更细腻的状态徽章和空状态文案
- 首页局部自动刷新与模块级重试
- 轻量动效，如呼吸高亮、柔和切换、悬浮反馈
- 更有个性的 MC 彩蛋图形元素

#### Could

- 日志页视图切换（时间线 / 紧凑列表）
- 更丰富的位置摘要可视化
- 更完整的插画系统
- 更多环境氛围型装饰与微互动

第一版计划应以 Must 为边界，Should 在时间允许时加入，Could 不作为首轮交付前提。

## 实现单元与边界

为了让后续计划可执行，第一版前端拆分为以下独立实现单元：

### 1. 应用壳层与导航单元

职责：

- 提供全局页面框架
- 承载桌面与移动端导航
- 管理当前工作区切换与全局视觉背景

输入：

- 当前激活页面
- 全局加载或通知状态

输出：

- 稳定页面布局容器
- 可切换的主内容区域

### 2. 首页聚合数据单元

职责：

- 并行获取首页需要的状态、资源、账号、实例、日志摘要
- 维护模块级 loading / success / error 状态
- 组装首页展示所需推导字段

输入：

- 五个首页依赖接口响应

输出：

- 头部状态横幅数据
- 总览主卡数据
- 账号工作区摘要
- 实例摘要
- 日志摘要

边界说明：

- 首页聚合数据由页面级查询层统一拉取
- 展示型卡片与资源模块只消费规范化后的数据，不自行发请求
- 页面级查询层负责自动刷新、局部重试触发与错误聚合
- 视图组件只接收 view model，不直接依赖原始 API response shape

页面级查询层最小接口：

- `loadHome()`：首页初次加载
- `refreshHome()`：首页全量刷新
- `retryHomeModule(module)`：模块级重试
- `refreshAfterMutation(scope)`：写操作成功后的局部刷新

### 3. 账号管理与登录任务面板单元

职责：

- 展示账号列表
- 创建账号
- 发起和轮询 Microsoft 设备码登录
- 清晰表达单会话替换规则

输入：

- 账号列表接口与登录相关接口

输出：

- 账号卡片集合
- 登录任务面板状态
- 成功 / 失败 / 替换反馈

边界说明：

- 该单元负责账号相关请求、轮询控制与状态机推进
- 展示组件只接收规范化账号状态和任务面板 view model

最小接口：

- `loadAccounts()`
- `createAccount(payload)`
- `startMicrosoftLogin(accountId)`
- `pollMicrosoftLogin()`
- `closeLoginPanel()`

### 4. 实例管理与动作卡片单元

职责：

- 展示实例列表
- 创建实例
- 提供启动、停止、重启动作
- 对动作中的 pending 和重复点击进行防护

输入：

- 实例列表接口与实例写操作接口

输出：

- 实例卡片集合
- 实例动作反馈
- 写操作后的局部刷新

边界说明：

- 该单元负责实例写操作与提交中状态
- 实例展示卡片不直接调用接口，只通过回调触发动作

最小接口：

- `loadInstances()`
- `createInstance(payload)`
- `startInstance(id)`
- `stopInstance(id)`
- `restartInstance(id)`

### 5. 资源与状态展示单元

职责：

- 呈现 CPU / 内存与整体状态概览
- 为首页和资源页复用同一套表达逻辑

输入：

- `GET /api/status`
- `GET /api/resources`

输出：

- 状态条、资源卡、占比与舒适度描述

边界说明：

- 该单元负责展示，不负责自行请求首页数据
- 在资源监控独立页面中，也优先复用同一展示模型和组件接口
- 自动刷新与重试仍由页面级查询层控制

最小接口：

- 输入统一为规范化后的状态与资源 view model
- 输出统一为横幅卡、资源卡、摘要标签等纯展示结果

## 资源监控页设计

资源监控作为一级工作区独立存在，但第一版保持轻量。

页面模块：

- 顶部资源摘要区：CPU、内存、采样时间
- 中部健康描述区：展示“舒适 / 繁忙 / 紧张”这类当前状态文案
- 底部说明区：提示这些数据是当前快照，不是完整历史分析

页面边界：

- 复用首页已有的资源展示组件和状态映射
- 不额外引入复杂时序图表系统
- 刷新策略与首页一致，以温和自动刷新和手动刷新为主
- 若资源接口失败，页面展示完整错误态与重试按钮
- 若资源接口成功但字段缺失，按首页缺失值规则回退显示

## 全局状态与共享协调层

为了支撑跨页一致性，第一版前端需要一个明确的全局状态与共享协调层。

职责：

- 保存当前激活导航与全局通知
- 保存应用级单例 Microsoft 登录任务状态
- 协调写操作后的局部刷新范围
- 协调跨页可见的成功 / 失败反馈

边界：

- 页面级查询层负责各页数据请求与 view model 组装
- 全局状态层不直接渲染页面，只提供共享状态和协调方法
- 首页与账号页共用同一个登录任务状态源

最小接口：

- `setActiveView(view)`
- `openLoginPanel()`
- `closeLoginPanel()`
- `setLoginTaskState(state)`
- `notifySuccess(message)`
- `notifyError(message)`
- `queueRefresh(scope)`

## 交付与构建边界

本设计默认延续仓库当前已经存在的前端交付约束：

- 前端源码位于 `frontend/`
- 前端开发构建产物输出到 `frontend/dist`
- 最终嵌入二进制的资源由现有 Go 打包流程筛选并复制到 `internal/webui/dist`

责任边界：

- 前端工程只负责生成标准 Vite 产物，不直接负责复制到 `internal/webui/dist`
- Go 打包步骤负责筛选允许嵌入的文件并产出最终可执行文件
- 前端请求始终走同源 `/api`，不在生产构建里写死额外 API 域名

开发环境约束：

- 本地开发可使用 Vite 开发服务器与后端联调
- 前端路由回退只作为 SPA 开发与嵌入后的页面能力存在，不改变 `/api` 路由语义
- 实现计划应包含前端构建与嵌入链路的最终验证步骤

验收与验证标准：

- 前端开发构建应成功产出 `frontend/dist`
- 现有 Go 打包入口为 `go run ./tools/packager`
- 打包后应由该流程把允许嵌入的前端产物复制到 `internal/webui/dist`
- 最终应验证嵌入构建后的程序既能响应 `/api`，也能返回前端页面
- 嵌入环境下访问任一前端页面路由并刷新时，不应白屏；应回退到前端入口页并继续由前端路由接管
- 前端静态资源路径需保持兼容当前嵌入式访问方式，默认以根路径部署，不额外引入子路径 base

### 6. 日志事件流单元

职责：

- 呈现最近操作日志
- 区分空状态、错误态与普通动态流

输入：

- `GET /api/logs/operations`

输出：

- 首页广播栏
- 独立日志页主列表

边界说明：

- 该单元负责日志列表的展示与空/错状态表达
- 是否刷新、何时刷新、如何重试由页面级查询层决定

最小接口：

- `loadLogs()`
- `retryLogs()`
- `mapLogEntriesToTimeline()`

这些单元之间只通过前端状态层和共享展示组件协作，不直接互相耦合业务细节。

## 响应式策略

桌面端是主体验，但移动端仍需可用。

### 桌面端

- 左侧导航 + 右侧主内容
- 首页可使用两列或 12 栅格布局
- 卡片允许较强的分层和留白

### 移动端

- 导航切换为底部 tab 或顶部抽屉
- 首页卡片改为单列堆叠
- 登录、创建、实例控制等关键动作仍需单手可达
- 不做复杂多栏表格，优先卡片化与折叠化

## 错误处理与状态反馈

虽然整体风格更轻松，但错误反馈必须清楚。

- 网络失败：直接提示请求失败，并提供重试入口
- 设备码登录超时 / 失败：显示明确失败文案，不用含糊可爱文案掩盖问题
- 创建实例失败：清楚显示失败原因，例如账号未登录或参数不合法
- 创建成功后：刷新受影响列表，并给出短暂成功反馈
- 轮询中网络抖动：显示“连接有点不稳，正在继续尝试”这一类中性提示，但不立即结束流程
- 列表接口超时：模块进入局部错误态，可单独重试
- 空响应：按空状态处理，而不是按错误处理

原则是：成功反馈可以更有温度，失败反馈必须直白。

## 测试与验证重点

前端实现后应至少覆盖以下验证点：

- 首页能正确并行加载状态、资源、账号、实例、日志摘要
- Microsoft 设备码登录流程能正确初始化、轮询、展示替换单活跃会话语义
- 账号创建、实例创建、实例启停动作能给出明确成功 / 失败反馈
- 空状态与错误状态都存在且不会阻塞下一步操作
- 在桌面与移动宽度下，导航与关键卡片可正常使用
- 构建产物兼容当前 `frontend/dist -> internal/webui/dist` 的嵌入与打包流程

## 实施建议

实现阶段应优先落地以下内容，而不是一开始追求所有细节满配：

1. 先建立整体视觉系统与应用外壳
2. 打通首页总览、账号流程、实例流程三条主路径
3. 再补资源监控与日志页的细节表达
4. 最后补充空状态插画、彩蛋动效和更细的文案打磨

这样可以保证第一版就已经“能用且有趣”，而不是把时间过多花在装饰层上。

## 结论

本次前端设计最终确定为一套“冒险基地风”的 gmcc 控制台：

- 它保留管理工具的清晰结构
- 它不追求严肃企业后台气质
- 它通过材质、色彩、文案、状态卡和局部叙事让产品更有个性
- 它优先服务账号管理、实例控制和状态感知三条核心主线

在不改变现有 API 结构的前提下，这套设计足以支持一个既可嵌入二进制、又具备强记忆点的前端控制台实现。
