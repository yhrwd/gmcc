# 前端同步协调层改造设计

## 背景

当前前端通过 `frontend/src/App.vue` 中的固定 `setInterval` 轮询首页、账号、实例与日志数据。这个实现有两个明显问题：

- 当网络较慢或后端响应时间变长时，上一轮请求尚未完成，下一轮轮询已经发出，导致同类请求重叠、状态更新顺序不稳定，甚至让界面进入“轮询出错但其实只是请求挂起”的状态。
- 创建、删除、登录、启停等操作虽然局部会手动触发一次刷新，但触发点分散在各个 store 内，刷新对象也不统一，导致有些操作完成后界面更新不及时，或者多个刷新互相碰撞。

用户期望的改造目标不是引入完整实时推送系统，而是在保留现有 HTTP API 的前提下，让前端和后端的状态同步更稳、更及时、更可预测。

## 目标

- 用统一同步协调层替换当前分散的裸轮询逻辑。
- 保证同一类资源刷新在任意时刻最多只有一个请求在飞。
- 当刷新期间再次收到刷新请求时，不并发叠加，而是在当前请求结束后自动补跑一次。
- 让创建、删除、修改账号/实例、实例启停、登录成功等操作在完成后能立即请求关联数据刷新。
- 在网络较差或后端短暂不稳定时，通过退避与降频避免前端持续叠加无效请求。
- 尽量复用现有 Pinia store 和 API client，不要求后端新增接口或改变现有 `/api` 语义。

## 非目标

- 本次不引入 SSE、WebSocket 或其他服务端主动推送通道。
- 本次不重写现有 store 的数据映射逻辑和界面结构。
- 本次不把所有请求都改造成全局可取消模型；优先解决重叠刷新与补刷一致性问题。
- 本次不改变 Microsoft 设备码登录流程的后端契约。

## 方案结论

采用“统一同步协调层 + 资源组串行调度 + 操作完成触发关联刷新”的方案。

核心思想如下：

- 保留 HTTP 拉取模式，但把刷新时机从各组件、各 store 自己管理，收拢到一个前端同步协调模块里。
- 每个资源组独立调度，串行执行，避免同组请求重叠。
- 当操作成功后，不再由业务 store 直接裸调多个 `loadXxx(true, true)`，而是通知协调层刷新受影响的数据组。
- 页面处于后台或连续失败时自动降频，恢复前台或请求成功后恢复正常节奏。

这个方案比只修 `setInterval` 更系统，但又不需要引入新的后端实时机制，适合作为当前阶段的稳定性修复与同步层收口方案。

## 设计原则

### 1. 单一职责

- store 继续负责“如何请求并保存数据”。
- 同步协调层负责“何时刷新、刷哪些、如何避免重叠”。
- 组件只负责生命周期接入与用户操作，不直接管理多个全局轮询器。

### 2. 最终一致优先

对于连续操作和慢网环境，优先保证界面尽快收敛到后端真实状态，而不是保证每个中间态都被完整展示。

### 3. 低侵入接入

尽量保留现有 `homeStore`、`accountsStore`、`instancesStore`、`logsStore` 的拉取函数签名，避免一次性重构过多 UI 代码。

### 4. 明确联动边界

每个业务操作都明确声明它会影响哪些资源组，避免“顺手全刷”和“漏刷”并存。

## 架构概览

新增一个前端同步协调模块，例如：`frontend/src/lib/sync.ts`。

该模块不存储业务数据，只维护同步任务状态，并注册一组资源刷新执行器（runner）。

### 主要参与者

- `sync coordinator`：统一调度中心，维护各资源组的定时、运行中状态、待补刷标记、失败计数和页面可见性影响。
- `resource runners`：对各 store 的刷新包装，负责实际调用 `loadHome(true, true)`、`loadAccounts(true, true)`、`loadInstances(true, true)`、`refreshLogs()`。
- `stores`：执行用户操作，并在成功后通知协调层刷新关联资源组。
- `App.vue`：应用入口，负责启动与停止协调层。

### 对外暴露方式

第一版协调层采用模块单例形式，由 `frontend/src/lib/sync.ts` 直接导出单例 `syncCoordinator`。

- `App.vue` 负责在应用生命周期内调用 `syncCoordinator.start()` 和 `syncCoordinator.stop()`。
- `accountsStore`、`instancesStore` 直接导入该单例，并在操作成功后调用 `syncCoordinator.requestNow(...)`。
- runner 注册也由 `App.vue` 在启动阶段完成，因为 runner 需要依赖已创建的 Pinia store 实例。

这样做的理由是：

- 当前前端入口清晰，使用单例能以最小改动打通组件和 store。
- 比 `provide/inject` 更适合被 store 直接调用。
- 比为此单独新增 Pinia 基础设施 store 更简单，也更容易在单元测试中替换或重置。

## 资源组划分

第一版同步协调层按以下资源组管理：

- `overview`：对应首页状态与资源摘要，执行 `homeStore.loadHome(true, true)`。
- `accounts`：对应账号列表，执行 `accountsStore.loadAccounts(true, true)`。
- `instances`：对应实例列表，执行 `instancesStore.loadInstances(true, true)`。
- `logs`：对应操作日志，执行 `logsStore.refreshLogs()`。

这样划分的原因是：

- `overview` 与 `instances` 有较强联动，但职责不同，不应绑成一个不可分组的大刷新。
- `logs` 时效性要求和网络容忍度与主状态不同，适合独立低频管理。
- `accounts` 的变更来源和刷新需求清晰，便于在登录成功、删除账号等事件后单独触发。

## 协调层职责与接口

同步协调层建议提供以下最小接口：

- `register(group, runner)`：为指定资源组注册 runner；同一组重复注册时以后一次为准；协调层未 `start()` 前允许注册，已 `start()` 后不允许再动态变更注册关系。
- `start()`：在四个资源组都已注册完成后启动协调层，注册页面可见性监听，并触发一次首轮刷新。
- `stop()`：清除定时器、停止后续调度、移除页面监听。
- `request(groups, reason)`：请求一组资源在下一次自然调度窗口执行；若该组当前正在运行，则只设置 `pending`，由当前请求结束后补跑；若当前空闲但已有自动定时器，则不额外插队，只保证这组不会错过下一轮刷新。
- `requestNow(groups, reason)`：显式要求指定组尽快执行；若当前空闲则立即启动一次刷新，并重新安排该组下一轮自动调度；若运行中则设置 `pending`，在当前请求结束后立刻补跑一次。业务操作成功后的补刷统一使用这个接口。
- `notifyVisibilityChange()`：内部根据页面前后台状态调整轮询频率。

`reason` 只用于日志、调试和测试断言，不参与业务逻辑分支。

为了减少实现分叉，第一版的调用约定是：

- 自动轮询和页面恢复前台后的补刷由协调层内部调用 `requestNow(...)`。
- 业务 store 的成功操作只调用 `requestNow(...)`。
- `request(...)` 预留给后续低优先级刷新场景，本次改造即便不被大量使用，也保留清晰语义。

补充约束如下：

- `request(groups, reason)` 的“不会错过下一轮刷新”定义为：若该组已有自动定时器，则保留原计划时间不变；若该组当前没有运行中请求也没有已安排定时器，则按该组当前有效间隔补一个新的自动定时器。
- `start()` 为幂等操作；重复调用时若协调层已经启动，则直接返回，不重复首刷、不重复注册监听。
- `stop()` 为幂等操作；重复调用时直接返回。
- 若某组在 `start()` 时尚未注册 runner，则 `start()` 应直接抛出配置错误，避免进入部分可用状态。

### 首轮刷新范围

`start()` 负责且只负责一次首轮刷新，范围固定为 `overview`、`accounts`、`instances`、`logs` 四个资源组。

- `App.vue` 只负责注册 runner 并调用 `start()`，不再单独触发任何首轮拉取。
- 首轮刷新允许四组并行执行。
- 如果其中某组首轮失败，不影响其他组继续完成。

## 单个资源组的状态模型

每个资源组维护如下运行时状态：

- `running`：当前是否已有请求在飞。
- `pending`：请求进行中期间是否又收到一次补刷请求。
- `timerId`：下一次自动调度的定时器。
- `failureCount`：连续失败次数，用于退避策略。
- `lastStartedAt`：最近一次执行开始时间，便于调试和测试。
- `intervalMs`：当前自动刷新间隔，基于前后台状态和失败次数动态计算。

这些状态只存在于前端内存中，不做持久化。

## 调度规则

### 0. 跨资源组并发策略

不同资源组允许并行执行，不做全局串行化，也不设置全局并发上限。

也就是说：

- `overview` 正在刷新时，不影响 `accounts` 或 `instances` 启动自己的刷新。
- 串行限制只在单个资源组内部生效。

这样可以保留首轮加载和前台恢复时的响应速度，也让每个资源组的失败、退避和补跑相互独立，更容易测试。

### 1. 串行执行

同一资源组在任意时刻最多只允许一个请求执行。

如果某组正在运行时再次收到刷新请求：

- 不发起第二个并发请求。
- 仅把该组标记为 `pending = true`。
- 当前请求结束后，如果检测到 `pending = true`，立即补跑一次，然后清除该标记。

这样可以解决慢网环境下固定间隔轮询导致的请求叠加问题。

### 2. 自动刷新采用“完成后再计划下一次”

自动刷新不再使用不关心请求是否结束的裸 `setInterval`。

每个资源组在本轮请求完成后，基于当前可见性状态和失败次数，重新计算并安排下一次刷新时间。这样可以保证：

- 请求再慢也不会产生同组重叠。
- 自动刷新节奏与实际网络状况更一致。

### 3. 页面可见性降频

页面位于后台时整体降频：

- `overview` / `accounts` / `instances` 默认前台间隔为 `8000ms`，后台间隔为 `20000ms`。
- `logs` 默认前台间隔为 `20000ms`，后台间隔为 `60000ms`。

当页面从后台回到前台时：

- 立即对主状态组补一次刷新。
- 然后恢复正常前台频率。

### 4. 连续失败退避

当某组连续失败时，下一次自动刷新间隔按失败次数逐步拉长，例如：

- 连续 1 次失败：保持该组当前默认间隔不变。
- 连续 2 次失败：该组下一次自动刷新间隔提升为默认间隔与 `15000ms` 的较大值。
- 连续 3 次及以上失败：该组下一次自动刷新间隔提升为默认间隔与 `30000ms` 的较大值。

成功一次后将 `failureCount` 清零，恢复默认频率。

这个规则不追求数学意义上的指数退避，但要求默认值固定、行为可预测、便于单元测试。

## runner 契约

每个资源组 runner 的统一契约为 `() => Promise<void>`。

- runner 成功完成时返回 resolved promise。
- runner 失败时通过 reject 抛出错误。
- 协调层只根据 runner 的 resolve / reject 更新 `failureCount`、`pending` 消费和下一轮调度。
- store 内部原有的错误状态更新仍然保留，但不作为协调层判断成功或失败的依据。

这意味着：

- 由于当前 `homeStore.loadHome(true, true)`、`accountsStore.loadAccounts(true, true)`、`instancesStore.loadInstances(true, true)`、`logsStore.refreshLogs()` 现状会吞掉错误并仅更新 store 状态，本次实现必须补一个明确适配层。
- 推荐做法是：将这些 store 拉取方法改为在失败时 `throw`，同时继续维护原有 store 错误状态；这样 runner 可以直接透传它们的 promise。
- 如果实现中发现直接修改 store 抛错会影响现有测试，也允许 runner 采用包装方式：调用 store 拉取后检查该次执行是否进入 error 状态，并在必要时主动 reject。
- 但无论采用哪种实现，最终对协调层暴露的 runner 都必须满足“成功 resolve、失败 reject”的统一契约。
- 单元测试可以通过伪 runner 明确模拟成功与失败，而不依赖 store 内部 state 侧写判断。

## 协调层生命周期与非法调用约定

### 生命周期模型

- `register()` 阶段：用于在启动前注册四个资源组的 runner。
- `start()` 阶段：协调层进入运行态，开始监听页面可见性并执行首轮刷新。
- `stop()` 阶段：协调层退出运行态，清除定时器，不再安排后续自动调度。
- `restart()` 场景：本次不单独提供 `restart()` 接口；应用若发生重新挂载，允许按 `register -> start -> stop` 的正常生命周期重新走一遍。

### 调用约束

- 未 `start()` 前调用 `request()` 或 `requestNow()`：先校验目标 group 是否都已注册；若存在未注册 group，则抛出配置错误；若 group 已注册，则直接忽略并返回，不安排请求。
- `stop()` 后调用 `request()` 或 `requestNow()`：先校验目标 group 是否都已注册；若存在未注册 group，则抛出配置错误；若 group 已注册，则直接忽略并返回，不安排请求。
- 对未注册的资源组调用 `request()` 或 `requestNow()`：无论当前是否处于运行态，都抛出配置错误，帮助尽早暴露接线问题。
- `start()` 后再次 `register()`：抛出配置错误，避免运行中动态换 runner。
- `stop()` 后注册表不保证可直接复用；下一次 `start()` 前必须重新对四个资源组执行 `register()`，以覆盖可能变化的 Pinia store 实例。
- 因此，`stop()` 后再次 `start()` 前若未重新注册完整四组 runner，应视为配置错误。

### 测试可重置性

由于协调层采用模块单例，本次实现需要提供测试专用重置能力，例如 `resetForTest()`，用于：

- 清空 runner 注册表
- 清空各资源组运行时状态
- 取消残留定时器
- 恢复未启动状态

该接口只用于单元测试，不在业务代码中使用。

## `overview` 组成功/失败判定

`overview` runner 对应 `homeStore.loadHome(true, true)`，而该方法内部会聚合状态与资源两个子请求。

为了让协调层退避规则明确，本次规格定义：

- 只要 `status` 或 `resources` 任一子请求失败，就将本次 `overview` runner 视为失败，并对协调层返回 reject。
- 若两个子请求都成功，则视为成功并返回 resolve。
- store 仍然保留各自的局部错误展示能力，例如只显示状态模块失败或资源模块失败。

这样处理的原因是：

- `overview` 组承担首页关键摘要职责，只要任一核心子请求失败，就说明这一组的全局数据未完整同步。
- 退避逻辑需要一个稳定且容易测试的单一判定标准。
- 局部展示与全局同步判定可以并存，不互相冲突。

## 业务操作与刷新联动

业务 store 在操作成功后，不直接分散调用自身或其他 store 的刷新函数，而是统一通知协调层刷新相关资源组。

本设计中的“修改账号/实例”专指未来或已有的账号编辑、实例配置编辑、启用状态切换等会改变后端持久化状态的成功操作。即使本仓库当前前端尚未完整暴露这些编辑入口，联动规则仍在规格中先定义清楚，避免目标与后续实现范围脱节。

### 推荐联动矩阵

- 创建账号：刷新 `accounts`
- 删除账号：刷新 `accounts`、`instances`、`overview`
- 修改账号：刷新 `accounts`、`instances`、`overview`
- Microsoft 登录成功：刷新 `accounts`、`overview`
- 创建实例：刷新 `instances`、`overview`
- 删除实例：刷新 `instances`、`overview`
- 修改实例：刷新 `instances`、`overview`
- 启动实例：刷新 `instances`、`overview`
- 停止实例：刷新 `instances`、`overview`
- 重启实例：刷新 `instances`、`overview`

### 关于日志刷新

`logs` 默认仍维持独立低频刷新，不对每个成功操作都强制立即补刷。理由如下：

- 日志不是当前问题的主要痛点。
- 每个操作都刷新日志会增加额外请求压力。
- 如果后续验证发现日志页面的“刚执行操作但看不到记录”体验仍然明显，再把关键操作成功后补刷 `logs` 作为第二阶段优化。

## 与现有登录轮询的关系

Microsoft 设备码登录面板目前已经采用“上一轮结束后再 `setTimeout` 下一轮”的串行轮询模式，相比全局 `setInterval` 更稳定。

本次改造不重写这套登录轮询机制，只做以下收口：

- 登录成功后不再只刷新账号列表，而是通过同步协调层请求刷新 `accounts` 和 `overview`。
- 登录失败、过期、取消等终态继续由原有面板逻辑负责展示。

也就是说，登录轮询仍是一个局部业务流程；全局同步协调层只接管其成功后的数据同步联动。

## 术语约定

- `主状态组`：固定指 `overview`、`accounts`、`instances` 三组，不包含 `logs`。
- `首轮刷新`：指协调层在 `start()` 成功后自动发起的首次全量资源组刷新。
- `补刷`：指某组在运行中收到新的刷新请求后，于当前请求结束后立即追加执行的一次刷新。

## 文件级改动方案

### `frontend/src/lib/sync.ts`

新增同步协调层实现，包含：

- 资源组类型定义
- 每组运行时状态定义
- runner 注册逻辑
- `register` / `start` / `stop` / `request` / `requestNow` 接口
- 自动调度、补跑、退避、前后台降频逻辑
- 测试专用重置能力

### `frontend/src/App.vue`

改为：

- 在组件挂载时启动同步协调层
- 注册 `overview`、`accounts`、`instances`、`logs` 对应 runner
- 在组件卸载时停止协调层

不再直接在 `App.vue` 中管理多个裸 `setInterval`。

由于 runner 闭包依赖当前 Pinia store 实例，每次应用挂载时都必须重新执行四组 `register()`，然后再 `start()`；不得依赖上一次挂载遗留的注册表。

如果实现中发现已有更上层的全局初始化入口可安全复用，也允许把 `start()` / `stop()` 接到那个入口，但前提是：

- 生命周期上仍然只能有一个活跃协调层实例。
- runner 注册时机仍在 Pinia store 可用之后。
- 不额外扩大本次改造范围去重构整个应用入口。

在当前仓库已知结构下，默认仍以 `App.vue` 作为接入点。

### `frontend/src/stores/accounts.ts`

改为：

- 创建账号成功后，通知协调层刷新 `accounts`
- 删除账号成功后，通知协调层刷新 `accounts`、`instances`、`overview`
- 修改账号成功后，通知协调层刷新 `accounts`、`instances`、`overview`
- Microsoft 登录成功后，通知协调层刷新 `accounts`、`overview`

store 自身仍然可以保留必要的本地 optimistic 更新和 toast 提示。

如果当前前端尚未暴露账号编辑入口，则“修改账号成功后刷新哪些组”的规则先作为接口约束写入规格，本次实现不需要为不存在的 UI action 新增页面功能，但若仓库中已有对应 action，则必须按该矩阵接入。

### `frontend/src/stores/instances.ts`

改为：

- 创建实例成功后，通知协调层刷新 `instances`、`overview`
- 删除实例成功后，通知协调层刷新 `instances`、`overview`
- 修改实例成功后，通知协调层刷新 `instances`、`overview`
- 启动 / 停止 / 重启成功后，通知协调层刷新 `instances`、`overview`

现有针对按钮忙碌态、本地卡片状态预更新的逻辑可以保留。

如果当前前端尚未暴露实例编辑入口，则“修改实例成功后刷新哪些组”的规则同样先作为未来约束保留；本次只要求接入当前代码里已存在的创建、删除、启动、停止、重启等操作。

### `frontend/src/stores/home.ts` 与 `frontend/src/stores/logs.ts`

原则上保留为“纯数据拉取实现”，不承担调度职责。

## 错误处理

### 网络慢但请求未失败

- 不视为错误，不主动终止。
- 同组后续刷新请求只会变成一次 `pending` 补刷。
- 当前请求完成后立即执行补刷，以尽快追平状态。

### 刷新请求失败

- 由现有 store 维持原有错误状态更新方式。
- 协调层记录失败次数，并按退避规则放慢下次自动刷新。
- 失败不会阻止业务操作按钮本身再次尝试。

### 操作成功但补刷失败

- 业务操作本身仍视为成功，toast 不回滚。
- 后续自动刷新会继续尝试恢复界面一致性。
- 测试中需要覆盖“操作成功 + 刷新失败”的场景，保证不会因为补刷失败把本地业务状态直接判成失败。

### 协调层停止后仍有未完成请求

- 本次不强制取消已发出的请求。
- 但请求结束后不再继续安排下一轮定时调度。

## 测试策略

### 单元测试：同步协调层

至少覆盖以下行为：

- 同一资源组不会并发执行两次
- 请求运行中再次触发会生成一次 `pending` 补跑
- 成功后重置失败计数
- 连续失败会拉长下一次自动刷新间隔
- `stop()` 后不再安排后续调度
- 页面从后台切回前台时会触发主状态组立即刷新
- 未 `start()`、已 `stop()`、未注册资源组等非法调用边界符合规格
- `resetForTest()` 能正确恢复单例初始状态

### Store 测试

更新账号和实例 store 测试，验证：

- 操作成功后请求了正确的资源组刷新
- 不再直接依赖分散的 `loadXxx(true, true)` 调用作为联动机制

### 现有行为回归

至少验证：

- 初次加载时 `overview`、`accounts`、`instances`、`logs` 四组 runner 都会被触发，且允许并行执行
- Microsoft 登录成功后账号状态能及时更新
- 实例启停后卡片最终状态能及时收敛到后端真实状态

## 风险与取舍

### 风险 1：协调层抽象过重

如果把协调层设计成过度通用框架，会让原本简单的 store 交互变得难懂。

应对方式：

- 第一版只支持当前四个资源组。
- 接口保持简单，只做本次需要的调度行为。

### 风险 2：业务操作与同步层耦合增加

store 需要知道哪些资源组受影响，这是一种显式耦合。

应对方式：

- 用固定联动矩阵控制范围。
- 保持“操作成功 -> 请求哪些组刷新”逻辑集中且可测试。

### 风险 3：未引入请求取消

极端情况下，慢请求仍可能长期占用一次刷新窗口。

应对方式：

- 第一阶段先解决重叠和补刷问题。
- 如果后续发现仍有明显“旧请求太慢导致新状态长期无法追平”的情况，再评估引入 `AbortController`。

## 实施顺序建议

1. 新增同步协调层，并为其补齐单元测试。
2. 在 `App.vue` 中接管全局自动刷新生命周期。
3. 改造 `accountsStore` 成功操作后的刷新联动。
4. 改造 `instancesStore` 成功操作后的刷新联动。
5. 运行前端测试，回归关键交互路径。

## 预期结果

改造完成后，前端与后端的协作应具备以下表现：

- 慢网环境下不再因固定轮询重叠导致同类请求堆积。
- 创建、删除、登录成功、实例启停后，界面会立即请求一次关联数据同步，而不是被动等待下一次全局轮询。
- 轮询节奏更可控，失败场景更温和，整体状态同步行为更容易理解和测试。
- 后续若要加入更复杂的同步方式（例如事件推送），也可以在保留这层协调接口的前提下逐步演进。
