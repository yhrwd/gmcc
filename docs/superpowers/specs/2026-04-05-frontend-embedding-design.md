# 前端嵌入二进制设计

## 背景

当前仓库 `gmcc` 是一个 API-only 的 Go 服务，Web 层基于 Gin，入口在 `cmd/gmcc/main.go`，HTTP 路由集中在 `internal/web/server.go`。现状是不内置前端静态资源，`README.md` 也明确说明当前运行模式为 API-only。

本次设计目标不是立即接入具体前端框架，而是先为未来的 Vue/Node 前端做好仓库结构、服务路由和打包链路准备，使后续可以把前端构建产物整合进单个二进制，同时保留当前 API 服务的稳定性。

## 目标

- 预留 `frontend/` 目录，供后续 Vue/Node 前端工程使用。
- 后端支持“有前端产物时提供页面、没有前端产物时继续提供 API”。
- 未来发行时可把前端静态资源嵌入 Go 二进制，实现单进程、一键启停。
- 打包流程使用 Go 编写脚本，负责筛选必要静态资源并生成最终二进制。
- `.gitignore` 明确忽略构建产物、前端依赖和打包输出，避免污染仓库。

## 非目标

- 本次不创建实际 Vue 工程，不引入 `package.json`、Vite、Webpack 等前端构建配置。
- 本次不设计具体前端页面、接口联调细节或 UI 结构。
- 本次不改变现有 `/api` 接口语义和已有业务逻辑。
- 本次不引入新的部署模式开关或复杂环境矩阵。

## 方案对比

### 方案 A：`frontend/` 放源码，构建产物复制到内部嵌入目录，再由 Go `embed` 打包（推荐）

- `frontend/` 只负责前端源码与配置。
- 新增内部嵌入目录，例如 `internal/webui/dist/`，只存放最终准备嵌入二进制的静态资源。
- Go 打包脚本在发行构建时，将未来前端构建目录中的必要资源复制到 `internal/webui/dist/`，再执行 `go build`。

优点：

- 源码目录与嵌入产物职责清晰。
- Go 代码不直接依赖 Node 工程目录，后续迁移构建工具时影响较小。
- 打包脚本更容易做资源白名单筛选，避免把开发资产带入二进制。

缺点：

- 打包流程需要多一步复制产物。

### 方案 B：直接嵌入 `frontend/dist`

- 未来前端构建直接输出到 `frontend/dist`。
- Go 代码直接 `embed frontend/dist`。

优点：

- 路径直观，构建链路更短。

缺点：

- Go 运行时代码直接耦合前端工程目录。
- 前端工程目录职责变杂，后续调整构建器、缓存目录或产物位置时更容易影响后端。

### 方案 C：运行时读取外部静态目录，不强制嵌入

- 后端优先读取磁盘静态目录，找不到则退回 API-only。

优点：

- 更灵活，替换前端时不必重编后端。

缺点：

- 不符合“整合到单个二进制、一键启停”的主要目标。
- 交付时更容易漏掉静态资源文件。

### 结论

采用方案 A。它在当前“先预留、后接入”的阶段最稳妥，也最符合仓库现有的显式边界和可预测行为风格。

## 目录结构设计

计划新增或约定以下目录：

- `frontend/`
  - 当前只放占位文件，例如 `.gitkeep` 或 `README.md`
  - 后续接入 Vue/Node 工程时，作为前端源码根目录
- `internal/webui/`
  - 放置前端嵌入资源访问逻辑
- `internal/webui/dist/`
  - 只放“准备被嵌入二进制”的静态资源产物
  - 由打包脚本生成，不手工维护
  - 为了满足 Go `embed` 的编译期约束，提交一个受控占位文件，例如 `.keep`
  - `internal/webui` 在运行时必须忽略该占位文件，且不能把它当作“前端已构建”的依据
- `tools/packager/` 或等价目录
  - 放 Go 打包脚本入口
- `build/` 或 `release/`
  - 放最终生成的二进制输出

目录职责约束：

- `frontend/` 永远表示前端源码与配置区。
- `internal/webui/dist/` 永远表示后端嵌入产物区。
- 不允许手工把非发行产物直接写入 `internal/webui/dist/`。
- `internal/webui/dist/.keep` 是唯一允许长期存在的受控占位文件，用于保证无前端产物时仍可通过编译。

## 架构与组件边界

### 1. Web 路由层：`internal/web`

职责：

- 保持现有 `/api` 路由与 handler 结构。
- 增加对前端静态资源和 SPA fallback 的分流能力。
- 在“无前端产物”时，对非 `/api` 请求返回明确提示。

边界：

- `internal/web` 负责 HTTP 请求分类与响应，不直接关心前端构建细节。
- 它通过一个前端资源访问层判断“是否有资源、是否命中文件、是否应该回 `index.html`”。

### 2. 前端资源访问层：`internal/webui`

职责：

- 暴露“是否存在前端资源”的能力。
- 提供静态文件查找能力。
- 提供 `index.html` fallback 读取能力。
- 屏蔽 `embed.FS`、路径归一化和资源存在性判断等细节。

建议对 `internal/web` 暴露一个可注入的最小接口语义，避免路由层感知底层文件系统细节。推荐形态是由 `internal/webui` 定义 `UIAssets` 接口，并提供默认实现构造函数，例如 `NewEmbeddedAssets()`。接口能力至少包括：

- `HasIndex()`：判断是否存在可用的前端入口页
- `LookupAsset(path string)`：判断请求路径是否命中真实静态资源，并返回统一的资源描述
- `OpenIndex()`：返回 `index.html`
- `IsAssetLikePath(path string)`：判断一个请求更像静态资源请求还是页面路由

接口输入约定：

- `LookupAsset` 与 `IsAssetLikePath` 的输入统一为 `c.Request.URL.Path`
- 输入不包含 query string 或 fragment
- 调用方不需要预先做额外 decode 或 `path.Clean`
- 允许传入带前导 `/` 的原始请求路径
- `internal/webui` 负责后续路径归一化和目录穿越防护

`LookupAsset` 的接口契约在本次设计中明确固定为：`(*AssetFile, error)`

- `nil, nil` 表示未命中资源
- 非 `nil, nil` 表示命中资源
- 非 `nil error` 表示异常

`AssetFile` 至少包含：

- `Name`
- `Content`
- `Size`
- `ModTime`
- `ContentType`

其中 `Content` 明确为 `[]byte`，不返回需要调用方关闭的流对象。这样可以让 `internal/web` 在 `GET` 与 `HEAD` 响应中使用统一逻辑，也避免把文件句柄生命周期暴露给路由层。

本次默认接受“将单个静态资源整体读入内存”这一实现约束。考虑到当前目标是管理面板类前端，这类构建产物规模通常可控；如果未来资源体积明显增大，再单独评估流式响应优化。

调用约定：

- 未命中不是错误
- 调用方无需自己推断 `Content-Type`
- `internal/web` 只消费稳定的资源描述，不直接操作嵌入文件系统

边界：

- `internal/webui` 不参与 API 业务逻辑。
- `internal/webui` 不负责前端构建，仅处理已准备好的嵌入资源。
- `internal/webui` 负责路径归一化、防目录穿越和嵌入资源存在性判断，这些逻辑应可独立测试。

### 3. 打包脚本：`tools/packager`

职责：

- 检查未来前端构建产物是否存在。
- 清理旧的 `internal/webui/dist/` 产物。
- 复制白名单资源到嵌入目录。
- 调用 `go build` 生成最终二进制。

边界：

- 打包脚本不承担前端工程初始化职责。
- 打包脚本只关心构建产物筛选和后端编译。

## 请求路由与数据流

### 路由判定顺序

服务器收到请求后，按以下顺序处理：

1. 如果路径属于 API 空间，即精确命中 `/api`、`/api/` 或 `/api/*`，继续按现有 Gin API 路由处理。
2. 如果不属于 API 空间，先按请求方法分类：只有 `GET` 和 `HEAD` 才允许进入静态资源或 SPA fallback 流程。
3. 对于非 `GET`/`HEAD` 的其他方法，若不属于 API 空间，则直接返回 `404`。
4. 对于 `GET`/`HEAD` 请求，检查该路径是否命中嵌入资源中的真实静态文件。
5. 若命中真实静态文件，则直接返回该文件内容。
6. 若未命中真实静态文件，但该路径被判定为“静态资源型路径”，则直接返回普通 404，不做 SPA fallback。
7. 若未命中静态文件，且该路径被判定为“页面路由型路径”，同时嵌入资源中存在 `index.html`，则返回 `index.html`，供 SPA 前端路由接管。
8. 若连 `index.html` 都不存在，则说明前端未构建，返回“当前仅提供 API”的降级响应。

### 关键行为约定

- API 空间（`/api`、`/api/`、`/api/*`）永远不走前端 fallback。
- SPA fallback 只适用于 `GET` 和 `HEAD`。
- 静态资源请求不会被错误回退到 `index.html`，避免 `assets/*.js` 被误判为页面路由。
- 前端页面路由如 `/instances/demo` 在有前端资源时应返回 `index.html`。
- 前端未构建时，非 `/api` 请求统一返回降级提示。

### 页面路由与静态资源的判定规则

为了让实现可执行，路由判定采用以下规则：

- 命中嵌入资源中的真实文件：按静态资源返回
- 未命中真实文件，但路径最后一段带文件扩展名，例如 `.js`、`.css`、`.png`、`.svg`、`.ico`、`.txt`、`.map`、`.json`：按静态资源请求处理，返回 404，不做 SPA fallback
- 未命中真实文件，但路径位于约定静态前缀下，例如 `/assets/`：按静态资源请求处理，返回 404
- 其余无扩展名路径，例如 `/`、`/instances`、`/instances/demo`：按页面路由处理，可回退到 `index.html`

补充约束：

- 本次设计默认不支持“带点号的 SPA 路由”作为兼容目标，例如 `/instances/v1.2`
- 本次设计也不为 `/.well-known/...` 这类特殊路径额外开放 SPA fallback 例外
- 如果未来前端明确需要这些路径形式，应在后续需求中单独扩展判定规则，而不是在本次先做宽松匹配

这条规则的目的，是让 SPA fallback 只覆盖“像页面”的路径，不覆盖“像文件”的路径。

## 前端未构建时的行为

当嵌入资源不存在或不包含 `index.html` 时：

- 服务仍正常启动。
- API 空间（`/api`、`/api/`、`/api/*`）保持可用。
- 非 `/api` 请求返回 `503 Service Unavailable`。
- 响应体统一使用简洁 HTML，明确表达：
  - 前端尚未构建
  - 当前服务仅提供 API

补充 HTTP 行为约定：

- `GET /` 与其他非 `/api` 页面请求，返回 `503`
- `HEAD` 请求沿用同样的状态码语义，只返回响应头不返回正文
- 不根据 `Accept` 头区分 JSON 或 HTML 版本，统一返回简洁 HTML；非浏览器客户端至少可以依赖状态码识别该能力不可用
- 其他非 `/api` 方法命中页面路径时，例如 `POST /foo`、`PUT /foo`、`DELETE /foo`，统一返回 `404`，不做页面 fallback
- 非 `/api` 的 `OPTIONS` 请求若未被其他中间件处理，也返回 `404`
- 降级页面响应的 `Content-Type` 固定为 `text/html; charset=utf-8`

选择 `503` 的原因：

- 语义上更准确，表示页面能力暂不可用，但服务本身未整体故障。
- 浏览器访问时可直接看到清晰提示。
- 不会让用户误以为前端已正确部署。

## 无前端产物时的编译策略

由于 Go `//go:embed` 在编译期要求匹配目标存在，本设计明确采用“受控占位文件”策略：

- `internal/webui/dist/` 提交一个固定占位文件，例如 `.keep`
- 嵌入规则覆盖该目录，使仓库在没有真实前端产物时也能成功编译
- 运行时由 `internal/webui` 明确忽略 `.keep`，并以“是否存在 `index.html`”作为前端是否可用的唯一判断依据

这样可以同时满足：

- 无前端产物时仍能编译 API-only 二进制
- 有前端产物时可正常嵌入真实静态资源
- 前端可用性判断不会被空目录占位文件误导

`.gitignore` 必须配套使用否定规则保留该占位文件，例如忽略 `internal/webui/dist/` 下其他产物，但显式保留 `internal/webui/dist/.keep`。

## 配置策略

本次不新增独立配置开关。采用自动检测策略：

- 有嵌入前端资源：自动提供前端页面和静态文件。
- 无嵌入前端资源：自动降级为 API-only + 页面访问 503 提示。

原因：

- 当前需求的重点是“默认好用”和“一键启停”，而不是多模式配置矩阵。
- 仓库现有风格也更偏向显式、可预测的默认行为。

如果后续确有需要，再考虑增加明确的 `web.ui.enabled` 或 `web.ui.mode` 配置项，但不纳入本次范围。

## 打包脚本设计

### 位置与形式

- 使用 Go 编写独立打包脚本。
- 推荐放在 `tools/packager/`，通过 `go run ./tools/packager` 执行。

### 脚本主流程

1. 解析输入参数或使用约定默认值。
2. 检查前端构建输出目录是否存在。
3. 若存在，则清理旧的 `internal/webui/dist/` 并复制白名单资源。
4. 若不存在，则打印提示并跳过前端资源准备。
5. 调用 `go build` 生成最终二进制到输出目录。

当 `-frontend-dist` 不存在时，脚本不能直接沿用旧的 `internal/webui/dist/` 真实产物，而是必须先把嵌入目录恢复到“仅保留 `.keep`”的干净状态，再继续构建 API-only 二进制。这样可以避免误把上次构建留下的前端资源带入本次产物。

### 参数与默认值

为避免行为含糊，脚本应定义明确的参数契约：

- `-frontend-dist`
  - 默认值：`frontend/dist`
  - 含义：前端构建产物输入目录
- `-embed-dist`
  - 默认值：`internal/webui/dist`
  - 含义：后端嵌入资源输出目录
- `-output`
  - 默认值：`build/gmcc.exe`（Windows）或 `build/gmcc`（非 Windows）
  - 含义：最终二进制输出路径

参数优先级采用“命令行参数高于默认值”的简单模型。本次不设计额外环境变量覆盖层。

### 脚本默认行为

- 未来如果发现前端构建产物，默认将其嵌入。
- 如果没有前端产物，不报错退出，继续生成 API-only 二进制。
- 如果前端输入目录存在，但缺少 `index.html`，视为“前端产物不可用”，清理嵌入目录到仅保留 `.keep`，并继续生成 API-only 二进制。
- 如果白名单筛选后没有留下 `index.html`，同样按 API-only 二进制处理，而不是失败退出。
- 输出日志要清楚说明当前构建结果是：
  - 包含前端资源的单文件二进制
  - 还是未包含前端资源的 API-only 二进制

### 嵌入目录清理规则

清理 `internal/webui/dist/` 时，脚本必须遵守以下规则：

- 可以删除旧的前端产物文件
- 不删除受控占位文件 `.keep`
- 如果采用“先清空再复制”的实现，结束前必须保证 `.keep` 被保留或恢复
- 即使前端输入目录不存在、筛选后为空，或复制过程失败，也不能让嵌入目录进入“既没有真实资源也没有 `.keep`”的不可编译状态

这样可以保证打包脚本在失败场景下仍不破坏后续构建稳定性。

## 资源筛选策略

打包脚本采用“白名单优先”的资源拷贝策略，避免把开发阶段无关文件打进二进制。

### 保留的资源

- 固定根文件：
  - `index.html`
  - `favicon.ico`
  - `favicon.svg`
  - `manifest.webmanifest`
  - `robots.txt`
- `assets/` 目录下的所有文件，递归复制

在复制 `assets/` 时，应用以下额外排除规则：

- 排除 `*.map`
- 排除隐藏临时文件
- 排除空目录

说明：如果未来前端工具输出了新的根级运行时必需文件，需要显式加入上述固定根文件白名单，而不是放宽为全量复制。

### 排除的资源

- `node_modules/`
- 前端源码目录，例如未来的 `src/`
- 原始公共资源目录，例如 `public/`
- sourcemap，例如 `*.map`
- 测试文件、缓存目录、临时文件、构建日志
- 任何不属于浏览器运行所需的开发资产

除上述白名单外，默认不复制其他根级文件。若未来前端确实新增必需运行时文件，应在脚本白名单中显式加入，而不是自动放开全量复制。

该策略满足“排除非必须资源”的要求，并控制最终二进制体积。

## 静态资源 HTTP 语义

静态资源命中时的响应行为应保持简单、可预测，并尽量复用标准库默认语义：

- `internal/webui` 负责产出稳定的资源元数据，包括 `ContentType`、`Size`、`ModTime`
- `internal/web` 负责把这些元数据写入 HTTP 响应
- `HEAD` 请求与 `GET` 使用相同的状态码和头部语义，但不返回正文
- 默认不在本次设计中增加自定义缓存策略；如使用标准库文件服务或等价逻辑，可接受其默认缓存行为
- 静态资源未命中且被判定为资源型路径时，返回普通 404，不回退到 `index.html`

本次明确不把以下 HTTP 语义纳入实现目标：

- `Range` 请求
- 条件请求协商，例如 `If-Modified-Since`
- 专门的缓存协商与强缓存策略定制

本次不把缓存优化纳入范围，避免在还没有真实前端工程前提前设计复杂静态资源策略。

## Git 忽略策略

`.gitignore` 需要补充前端与打包相关规则，但只忽略产物，不忽略占位目录或源码目录。

应忽略：

- `frontend/dist/`
- `frontend/node_modules/`

## 错误处理设计

- API 未命中：保持现有 JSON 404 风格，不受前端路由影响。
- 静态文件请求命中失败且被判定为资源型路径：返回普通 404。
- 页面路由在前端已嵌入时：返回 `index.html`，交由前端决定是否展示 404 页面。
- 页面路由在前端未构建时：返回 `503` 和简洁提示。
- 打包脚本在没有前端构建产物时：打印说明，不视为错误。
- 打包脚本在复制资源或执行 `go build` 失败时：返回明确错误并终止。
- `internal/webui` 在处理请求路径时必须做路径归一化，并拒绝目录穿越类路径。

## 测试策略

至少补充以下测试：

- `internal/webui` 包测试：
  - `index.html` 存在性判断正确
  - 静态资源查找正确
  - 带扩展名路径与无扩展名路径分类正确
  - 路径归一化与目录穿越防护正确
- `internal/web` 包测试：
  - `/api/*` 继续返回原有 API 响应
  - 有嵌入资源时，根路径返回 `index.html`
  - 有嵌入资源时，静态资源路径返回真实文件
  - 无嵌入资源时，非 `/api` 路径返回 `503`
- 打包脚本测试或最小行为验证：
  - 至少提供 `go test ./tools/packager -v` 的自动化验证，不仅依赖人工检查
  - 无前端构建产物时仍可生成后端二进制
  - 清理嵌入目录后仍会保留或恢复 `.keep`
  - 输入目录存在但缺少 `index.html` 时，回退为 API-only 构建
  - 资源筛选不会把明显无关文件带入嵌入目录
  - 只复制 `index.html`、`assets/**` 和固定根文件白名单，不复制其他根级文件

完成实现后建议执行：

- `go fmt ./...`
- `go test ./internal/web`
- `go test ./...`
- `go build -o gmcc.exe ./cmd/gmcc`

## 实施顺序建议

1. 新增 `frontend/` 占位目录和 `.gitignore` 规则。
2. 新增 `internal/webui` 资源访问层。
3. 调整 `internal/web/server.go` 路由分流逻辑。
4. 编写 Go 打包脚本。
5. 补充测试与 README 说明。

这个顺序可以先建立目录与边界，再逐步引入前端资源能力，最后收尾打包与文档。

## 成功标准

- 仓库中存在明确的 `frontend/` 预留目录。
- 服务在无前端资源时仍可正常提供 `/api`。
- 服务在有前端嵌入资源时可同时提供页面与 `/api`。
- 未来前端构建产物可通过 Go 打包脚本筛选后嵌入二进制。
- 构建产物、依赖目录和输出目录被 `.gitignore` 正确忽略。
- 新行为有对应测试覆盖，且不破坏当前 API 功能。

## README 口径

README 应从“当前服务是 API-only 运行模式，不再内置前端静态资源”调整为更准确的表述：

- 默认运行时仍可以是 API-only
- 但服务已支持在发行构建中可选嵌入前端静态资源
- 当未包含前端资源时，服务继续提供 `/api`，页面请求返回 `503`

这样既不误导当前使用者，也能准确反映新增能力。
